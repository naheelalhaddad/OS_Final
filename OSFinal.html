Chapter 2 — Operating-System Structures & System Calls

Boot Sequence

BIOS/UEFI → POST (hardware self-test) → finds a bootable device → Boot loader runs and finds the kernel → Kernel loads & initializes drivers/memory → Kernel starts the init process (PID 1) → init starts services, daemons, and the GUI.
تسلسل الإقلاع: الـ BIOS/UEFI يفحص الهاردوير (POST) ثم يبحث عن جهاز إقلاع، فيعمل محمّل الإقلاع الذي يجد النواة ويحمّلها، ثم تهيّئ النواة السائقات والذاكرة وتبدأ أول عملية مستخدم (init، رقمها 1) التي تشغّل بقية الخدمات وواجهة المستخدم.

OS Services

Help the user: User interface (GUI/CLI/touch), program execution, I/O operations, file-system manipulation, communications, error detection.
Help the system: resource allocation, logging/accounting, protection & security.
خدمات تساعد المستخدم: واجهة المستخدم، تنفيذ البرامج، عمليات الإدخال/الإخراج، التعامل مع الملفات، الاتصال بين العمليات، اكتشاف الأخطاء. خدمات تساعد النظام: توزيع الموارد، تسجيل الاستخدام، والحماية والأمان.

System Calls

A system call is how a user program asks the kernel to do something it can't do itself (open a file, create a process). Usually written in C/C++ or assembly.
استدعاء النظام هو الطريقة التي يطلب بها برنامج المستخدم من النواة تنفيذ عملية لا يستطيع تنفيذها بنفسه، مثل فتح ملف أو إنشاء عملية جديدة.

cp source_file.txt destination_file.txt

cp source_file.txt destination_file.txt
This single shell command triggers system calls internally: open(), read(), write(), close().

Worked Example — Steps of a System Call SOLVED

EN
read(fd, buffer, nbytes) reads n bytes from a file into a buffer. 11 steps: 
1-3:  push parameters (fd, buffer, nbytes) onto the stack
4:    call the library routine "read"
5:    place the system-call code number for read into a register
6:    TRAP into the operating system (switch to kernel mode)
7-8:  OS saves user state, calls the correct handler routine
9-10: handler finishes, control returns to the user program
11:   pop the parameters off the stack
AR
استدعاء read(fd, buffer, nbytes) يقرأ عدد بايتات من ملف لداخل buffer، وله 11 خطوة: دفع المعاملات على المكدس (1-3)، استدعاء روتين المكتبة (4)، وضع رمز الاستدعاء في سجل (5)، حدوث Trap للانتقال لنمط النواة (6)، حفظ النواة لحالة المستخدم واستدعاء المعالج المناسب (7-8)، عودة السيطرة للمستخدم (9-10)، وسحب المعاملات من المكدس (11).

Categories of System Calls

  1. Process control — create/terminate, load/execute, wait, allocate/free memory.
  2. File management — create/delete/open/close/read/write, get/set attributes.
  3. Device management — request/release device, read/write, get/set attributes.
  4. Information maintenance — get/set time, system data, process attributes.
  5. Communications — create/delete connection, send/receive messages.
  6. Protection — set/get permissions, allow/deny access.
ستة أنواع لاستدعاءات النظام: التحكم بالعمليات، التحكم بالملفات، التحكم بالأجهزة، صيانة المعلومات، الاتصالات، والحماية.

Chapter 3 — Processes

Process Memory Layout

  • Text — the executable code (fixed).
  • Data — global variables.
  • Heap — dynamically allocated memory at run time.
  • Stack — function parameters, return addresses, local variables. Each call pushes an activation record; popped on return.
تتكون العملية في الذاكرة من 4 أقسام: قسم النص (الكود)، قسم البيانات (المتغيرات العامة)، الكومة Heap (ذاكرة تُحجز ديناميكيًا)، والمكدس Stack (بيانات مؤقتة كمعاملات الدوال، تُدفع سجلة تنشيط جديدة عند كل استدعاء وتُسحب عند العودة).

Process States

New → Ready → Running → (Waiting ↔ Ready) → Terminated.
حالات العملية: جديدة، جاهزة، قيد التنفيذ، منتظرة (لحدث كـI/O)، ومنتهية.

Process Control Block (PCB)

Holds: process state, program counter, CPU registers, scheduling info, memory-management info, accounting info, I/O status info.
كتلة التحكم بالعملية (PCB) تحفظ: حالة العملية، عداد البرنامج، السجلات، معلومات الجدولة، إدارة الذاكرة، المحاسبة، وحالة الإدخال/الإخراج.

Scheduling Queues & Schedulers

Job queue (all processes) → Ready queue (in memory, ready) → Device queue (waiting for I/O).
Short-term scheduler: picks next process to run — fast, frequent.
Long-term scheduler: decides which jobs enter memory — slow, rare.
Medium-term scheduler: swaps a process out to reduce multiprogramming.
طابور المهام يحوي كل العمليات، طابور الجاهزية يحوي العمليات الجاهزة بالذاكرة، وطابور الأجهزة يحوي العمليات المنتظرة لجهاز. المجدول قصير المدى يختار العملية التالية بسرعة، المجدول طويل المدى يحدد أي المهام تدخل الذاكرة ويعمل ببطء، والمجدول متوسط المدى يبدّل عمليات للخارج عند امتلاء الذاكرة.

Context Switch & Process Creation/Termination

Context switch = save old process state, load new process state. Pure overhead. A parent creates children (PID-identified, forming a tree). Termination via exit(); parent collects status via wait(); parent can force-kill via abort() (e.g. child exceeded resources, or parent itself exits → cascading termination).
تبديل السياق يحفظ حالة العملية القديمة ويحمّل الجديدة، وهو عبء زائد بدون عمل مفيد. العملية الأم تنشئ أبناء (شجرة عمليات، كل واحد له PID). الإنهاء عبر exit()، وتجمع الأم الحالة عبر wait()، وتستطيع إنهاء الابن بـabort() (تجاوز الموارد، أو انتهاء الأم نفسها = إنهاء متسلسل).

Interprocess Communication (IPC)

Shared memory — common memory region read/written by both. Message passing — direct message exchange, no shared region.
الذاكرة المشتركة: منطقة ذاكرة تتشارك فيها العمليات. تمرير الرسائل: تبادل رسائل مباشرة بدون ذاكرة مشتركة.

Chapter 4 — Threads & Concurrency

What is a Thread

A thread has its own ID, program counter, registers, and stack — but shares code, data, and OS resources with other threads of the same process.
الخيط له معرّف وعداد برنامج وسجلات ومكدس خاص به، لكنه يتشارك الكود والبيانات وموارد النظام مع باقي خيوط نفس العملية.

Benefits of Multithreading

Responsiveness, Resource Sharing, Economy (cheaper than processes), Scalability (parallel on multiple cores).
الفوائد: الاستجابة، مشاركة الموارد، الاقتصادية، وقابلية التوسع على أنوية متعددة.

Multithreading Models

Many-to-One — no true parallelism, one block stalls everything. One-to-One — true parallelism, costly thread creation. Many-to-Many — best of both.
Many-to-One: لا يوجد توازي حقيقي. One-to-One: توازي حقيقي لكن تكلفة إنشاء أعلى. Many-to-Many: يجمع مزايا النموذجين.

Concurrency vs Parallelism

Concurrency: progress on multiple tasks over overlapping time (may just interleave on one core). Parallelism: tasks literally execute at the same instant on separate cores.
التزامن: تقدم عدة مهام خلال فترات متداخلة (قد تتبادل المعالج فقط). التوازي: تنفيذ فعلي بنفس اللحظة على أنوية مختلفة.

Amdahl's Law

Speedup = 1 / ( S + (1 − S)/N )
S = serial fraction of the task, N = number of cores.
S هي النسبة التسلسلية التي لا يمكن تسريعها، N عدد الأنوية.

Worked Example — Amdahl's Law SOLVED

EN
Matrix multiply: Step1 (parallelizable) = 80%, Step2 (serial) = 20%. Quad-core (N=4): 
S = 0.20,  N = 4
Speedup = 1 / (0.20 + 0.80/4) = 1 / 0.40 = 2.5
Only 2.5× faster despite 4 cores — the serial 20% caps the gain.
AR
الخطوة الأولى 80% قابلة للتوازي والثانية 20% تسلسلية. مع 4 أنوية: التسريع = 1 ÷ (0.20 + 0.80/4) = 2.5 مرة فقط، لأن الجزء التسلسلي يحدّ من الفائدة القصوى.

Chapter 5 — CPU Scheduling (Part 1 + Part 2)

The Dispatcher

The dispatcher gives control of the CPU to the process selected by the short-term scheduler. It performs: context switching, switching to user mode, and jumping to the proper location to restart the program. Dispatch Latency: Time taken to stop one process and start another.
الـ **Dispatcher** هو الموديل البرمجي الذي يمنح التحكم بالمعالج للعملية المحددة بواسطة المجدول قصير المدى. وظائفه: تبديل السياق، الانتقال لنمط المستخدم، والقفز للموقع الصحيح لإعادة تشغيل البرنامج. **Dispatch Latency** هو الوقت المستغرق لإيقاف عملية وتشغيل أخرى (وهو عبء زائد Overhead).

Basic Concepts

A process alternates CPU bursts and I/O bursts. CPU-bound = few long bursts. I/O-bound = many short bursts. Nonpreemptive: once started, runs until done or it blocks itself. Preemptive: OS can take the CPU away (timeout, higher-priority arrival).
العملية تتبادل انفجارات معالج وانفجارات إدخال/إخراج. غير الاستباقية: تستمر حتى تنتهي أو تنتظر بنفسها. الاستباقية: يمكن سحب المعالج منها قسرًا.

Scheduling Criteria

CPU utilization (max), Throughput (max), Turnaround time (min), Waiting time (min), Response time (min).
استخدام المعالج (زيادة)، الإنتاجية (زيادة)، زمن الدوران (تقليل)، زمن الانتظار (تقليل)، زمن الاستجابة (تقليل).

1. First-Come First-Served (FCFS)

Simple FIFO, nonpreemptive. Problem: convoy effect — short processes wait behind long ones.
طابور FIFO بسيط وغير استباقي. مشكلته الأساسية تأثير القافلة: العمليات القصيرة تنتظر خلف الطويلة.

2. Shortest Job First (SJF) / Shortest Remaining Time First (SRTF)

Runs the shortest next CPU burst first. Nonpreemptive: runs to completion once started. Preemptive (SRTF): a new shorter process preempts the running one. Minimizes average waiting time if bursts are known.
تُنفّذ العملية ذات أقصر انفجار معالج تالٍ أولًا. النسخة غير الاستباقية تكمل التنفيذ كاملًا. النسخة الاستباقية SRTF تُقاطع العملية الجارية إذا وصلت عملية أقصر.

3. Priority Scheduling

CPU → highest priority (smaller number = higher priority). Problem: starvation. Fix: aging (raise priority the longer a process waits).
المعالج يُمنح للأولوية الأعلى (الرقم الأصغر = أولوية أعلى). المشكلة: التضوّر. الحل: الشيخوخة (رفع الأولوية كلما طال الانتظار).

4. Round Robin (RR)

FCFS + preemption with a fixed time quantum q. Larger q → behaves like FCFS. Smaller q → more context-switch overhead.
مثل FCFS لكن استباقية بكوانتم زمني ثابت. كوانتم كبير ⇐ يتصرف كـFCFS، كوانتم صغير ⇐ كلفة تبديل سياق أعلى.

5. Multilevel Queue & 6. Multilevel Feedback Queue (MLFQ)

Multilevel queue: processes permanently split into queues (e.g. interactive vs batch), each with its own algorithm. MLFQ: like multilevel queue, but a process can move between queues — demoted if it uses too much CPU, promoted (aging) if it waits long.
الطابور متعدد المستويات: تقسيم دائم للعمليات لطوابير منفصلة. الطابور متعدد المستويات المرتد (MLFQ): يمكن للعملية الانتقال بين الطوابير، تُخفَّض إن استخدمت معالجًا كثيرًا، وتُرفَّع إن انتظرت طويلًا.

Chapter 6 — Process Synchronization (Full Code Toolkit)

Race Condition

Happens when multiple processes/threads touch shared data concurrently, and the result depends on execution timing.
تحدث عندما تصل عدة عمليات أو خيوط لبيانات مشتركة بنفس الوقت، وتعتمد النتيجة على ترتيب التنفيذ بالضبط.

Producer–Consumer (shared buffer + counter)

// producer process
while (true) {
    while (counter == BUFFER_SIZE) ;   /* buffer full, wait */
    buffer[in] = next_produced;
    in = (in + 1) % BUFFER_SIZE;
    counter++;
}

// consumer process
while (true) {
    while (counter == 0) ;             /* buffer empty, wait */
    next_consumed = buffer[out];
    out = (out + 1) % BUFFER_SIZE;
    counter--;
}

Why counter++ / counter-- races

counter++  compiles to:        counter--  compiles to:
  r1 = counter                   r2 = counter
  r1 = r1 + 1                    r2 = r2 - 1
  counter = r1                   counter = r2

// counter starts at 5, both run "at the same time":
S0: producer  r1 = counter         // r1 = 5
S1: producer  r1 = r1 + 1          // r1 = 6
S2: consumer  r2 = counter         // r2 = 5
S3: consumer  r2 = r2 - 1          // r2 = 4
S4: producer  counter = r1         // counter = 6
S5: consumer  counter = r2         // counter = 4   <- WRONG, should be 5
Fix: wrap counter++ / counter-- inside a critical section (mutex / semaphore) so the 3 steps run atomically.

Critical Section — 3 Required Properties

Mutual Exclusion (only 1 process in CS at a time), Progress (entry decision can't be postponed forever), Bounded Waiting (a cap on how many others can enter before you get your turn).
الاستثناء المتبادل (عملية واحدة فقط بالمنطقة الحرجة)، التقدم (لا يجوز تأجيل قرار الدخول للأبد)، والانتظار المحدود (حد أعلى لدخول عمليات أخرى قبل دورك).

Code 1 — test_and_set()

An atomic hardware instruction: reads the old value AND sets it to true, in one uninterruptible step. Used to build a spinlock.
تعليمة هاردوير ذرية: تقرأ القيمة القديمة وتضبطها على true بخطوة واحدة غير قابلة للمقاطعة، تُستخدم لبناء قفل دوّار (Spinlock).

test_and_set.c

// Hardware-level atomic instruction (this whole function is one indivisible step)
boolean test_and_set(boolean *target) {
    boolean rv = *target;
    *target = true;
    return rv;
}

// Using it to protect a critical section
boolean lock = false;

while (true) {
    while (test_and_set(&lock))
        ;                     /* busy wait: lock was already true */

    /* critical section */

    lock = false;             /* release */

    /* remainder section */
}

Code 2 — compare_and_swap()

Atomic instruction: swaps a value only if it currently equals an expected value.
تعليمة ذرية: تُبدّل القيمة فقط إذا كانت تساوي قيمة متوقعة.

compare_and_swap.c

// Hardware-level atomic instruction
int compare_and_swap(int *value, int expected, int new_value) {
    int temp = *value;
    if (*value == expected)
        *value = new_value;
    return temp;
}

// Using it to protect a critical section
int lock = 0;

while (true) {
    while (compare_and_swap(&lock, 0, 1) != 0)
        ;                     /* busy wait: someone else holds it */

    /* critical section */

    lock = 0;                 /* release */

    /* remainder section */
}

Code 3 — Mutex Lock

A higher-level software lock built on top of hardware atomics. Must acquire() before the critical section, release() after.
قفل برمجي أعلى مستوى، مبني فوق تعليمات الهاردوير الذرية. يجب حجزه قبل المنطقة الحرجة وتحريره بعدها.

mutex.c

typedef struct {
    boolean available;   // true = free, false = locked
} lock_t;

void acquire(lock_t *lock) {
    while (!lock->available)
        ;                     /* busy wait */
    lock->available = false;
}

void release(lock_t *lock) {
    lock->available = true;
}

// Usage
lock_t mutex = { .available = true };

while (true) {
    acquire(&mutex);

    /* critical section */

    release(&mutex);

    /* remainder section */
}

Code 4 — Peterson's Solution (2 processes only)

Pure software, works for exactly 2 processes. Uses shared turn and flag[2].
حل برمجي بالكامل، يعمل لعمليتين فقط. يستخدم متغيرين مشتركين: turn و flag[2].

peterson.c

#define TRUE  1
#define FALSE 0

int turn;
int flag[2] = { FALSE, FALSE };

// Process Pi calls this before the critical section (i = 0 or 1)
void enter_critical_section(int i) {
    int j = 1 - i;            // the other process
    flag[i] = TRUE;
    turn = j;
    while (flag[j] == TRUE && turn == j)
        ;                     /* busy wait */
}

// Process Pi calls this after the critical section
void leave_critical_section(int i) {
    flag[i] = FALSE;
}

// Usage inside process Pi
void process_Pi(int i) {
    while (TRUE) {
        enter_critical_section(i);

        /* critical section */

        leave_critical_section(i);

        /* remainder section */
    }
}
If both processes call enter_critical_section() at almost the same time, "turn" gets written by both — only the last write survives, and that decides who actually goes first.

Code 5 — Semaphore (busy-wait version)

A semaphore is an integer accessed only through wait() (decrement) and signal() (increment). Binary semaphore (0/1) behaves like a mutex. Counting semaphore (any range) controls multiple-instance resources.
السيمافور متغير عددي يُستخدم فقط من خلال wait() (تنقيص) وsignal() (زيادة). السيمافور الثنائي (0 أو 1) يتصرف كقفل متبادل. السيمافور العدّاد (أي نطاق) يتحكم بموارد متعددة النسخ.

semaphore_busy_wait.c

typedef int semaphore;

void wait(semaphore *S) {
    while (*S <= 0)
        ;                     /* busy wait: wastes CPU cycles */
    (*S)--;
}

void signal(semaphore *S) {
    (*S)++;
}

// Usage: protecting a critical section
semaphore mutex = 1;          // binary semaphore

while (true) {
    wait(&mutex);

    /* critical section */

    signal(&mutex);

    /* remainder section */
}

Code 6 — Waitless (Blocking) Semaphore

Avoids busy-waiting. Instead of looping, a process that must wait is blocked (placed in the semaphore's own queue, PCB state set to "waiting") and only restarted via wakeup() when another process calls signal().
يتجنب الانتظار المُشغِل. بدلًا من التكرار، تُحجَب العملية المنتظرة (تُوضَع في طابور خاص بالسيمافور، وتتحول حالتها بـPCB إلى "منتظرة")، وتُستدعى مجددًا عبر wakeup() عند تنفيذ signal() من عملية أخرى.

semaphore_blocking.c

typedef struct {
    int value;
    struct process *list;     // queue of PCBs waiting on this semaphore
} semaphore;

void wait(semaphore *S) {
    S->value--;
    if (S->value < 0) {
        add_this_process_to(S->list);   // change this process's PCB state to "waiting"
        block();                        // suspend; give CPU to the scheduler
    }
}

void signal(semaphore *S) {
    S->value++;
    if (S->value <= 0) {
        struct process *P = remove_one_process_from(S->list);
        wakeup(P);                       // move P from "waiting" to "ready"
    }
}
block() suspends the calling process and hands control to the scheduler. wakeup(P) moves process P from the waiting queue back to the ready queue.

Synchronization Problems

Busy waiting (wastes CPU), Starvation (a process never gets the resource), Deadlock (processes wait for each other forever).
الانتظار المُشغِل (يهدر دورات المعالج)، التضوّر (عدم حصول عملية على المورد أبدًا)، والانغلاق (انتظار العمليات لبعضها للأبد).

Exercises — Race Conditions

Bank account: deposit() / withdraw()

If husband withdraws and wife deposits at the same time, both might read the same starting balance before either writes back — one update is lost. Fix: wrap the read-modify-write of the balance in a critical section (mutex/semaphore).
إن سحب الزوج وأودعت الزوجة بنفس الوقت، قد يقرأ كلاهما الرصيد القديم نفسه قبل الكتابة، فيضيع أحد التحديثين. الحل: حماية القراءة والتعديل والكتابة بمنطقة حرجة.

Online auction: bid()

void bid(double amount) {
    if (amount > highestBid)
        highestBid = amount;
}
Two bidders calling bid() concurrently can both read the old highestBid, both pass the check, then the smaller bid can overwrite the larger one. Fix: make the read-compare-write atomic.
قد يقرأ مزايدان القيمة القديمة لـhighestBid معًا، فتُكتب القيمة الأصغر فوق الأكبر خطأ. الحل: جعل القراءة والمقارنة والكتابة عملية ذرية واحدة.

Parallel array sum

for (j = 1; j <= log2(N); j++) {
    for (k = 1; k <= N; k++) {
        if ((k + 1) % pow(2, j) == 0) {
            values[k] += values[k - pow(2, j - 1)];
        }
    }
}
No race condition — at every level j, each core reads/writes a disjoint index, so no two cores touch the same memory location at the same step.
لا توجد حالة تسابق، لأنه في كل خطوة j تتعامل كل نواة مع فهرس مختلف ولا يحدث تداخل بالكتابة على نفس الموقع.

Chapter 7 — Deadlocks

The 4 Necessary Conditions

  1. Mutual exclusion — only one process per resource at a time.
  2. Hold and wait — holding a resource while waiting for more.
  3. No preemption — resources released only voluntarily.
  4. Circular wait — P0→P1→P2→...→Pn→P0, each waiting on the next.
All 4 must hold together for a deadlock to occur.
الاستثناء المتبادل (مورد واحد لعملية واحدة)، الحجز والانتظار (حجز مورد وانتظار آخر)، عدم السلب (تحرير طوعي فقط)، والانتظار الدائري (حلقة انتظار مغلقة). الانغلاق يحتاج الأربعة معًا.

Resource-Allocation Graph

Circles = processes, Rectangles = resources. Request edge Pi→Rj, Assignment edge Rj→Pi. No cycle → no deadlock. Cycle + single instance per resource → deadlock. Cycle + multiple instances → only possible deadlock.
دوائر = عمليات، مستطيلات = موارد. سهم الطلب Pi→Rj، سهم التخصيص Rj→Pi. لا دورة = لا انغلاق. دورة + نسخة واحدة لكل مورد = انغلاق حتمي. دورة + نسخ متعددة = احتمال انغلاق فقط.

Worked Example — Cycle Detection SOLVED

EN
P1 holds R1, requests R2. P2 holds R2, requests R1. Edges: R1→P1, P1→R2, R2→P2, P2→R1. Path: P1→R2→P2→R1→P1 — a cycle. Both resources have 1 instance → this is a real deadlock: P1 waits forever for R2, P2 waits forever for R1.
AR
P1 تملك R1 وتطلب R2، وP2 تملك R2 وتطلب R1، فتتكون دورة. بما أن كل مورد له نسخة واحدة، فهذا انغلاق حقيقي: كل عملية تنتظر المورد الذي تملكه الأخرى للأبد.

Handling Methods

Prevention/Avoidance (never enter a deadlock state), Detection & Recovery (let it happen, then fix), Ignorance (assume it never happens — most general OSes).
الوقاية/التجنب، الاكتشاف والتعافي، أو التجاهل (تفترضه أغلب أنظمة التشغيل العامة).

Deadlock Prevention

Break one of the 4 conditions: allow sharable resources to bypass mutual exclusion; force requesting all resources upfront (hold-and-wait fix, costs utilization/starvation); force releasing everything if a new request can't be met (no-preemption fix); impose a total ordering on resource types (circular-wait fix).
كسر أحد الشروط الأربعة: السماح بمشاركة الموارد القابلة للمشاركة، إلزام حجز كل الموارد مسبقًا (يقلل الاستخدام ويُسبب تضوّرًا)، إلزام تحرير كل الموارد عند تعذر طلب جديد، أو فرض ترتيب ثابت لأنواع الموارد.

Deadlock Avoidance — Safe State

A state is safe if some order of ALL processes exists where each one's remaining need can be met by what's available + what earlier processes in that order will release. Safe → no deadlock. Unsafe → only a possibility.
الحالة أمنة إذا وُجد ترتيب لكل العمليات بحيث تُلبَّى حاجة كل واحدة بالموارد المتاحة + ما ستحرره العمليات السابقة لها. الحالة الأمنة تعني عدم وجود انغلاق، وغير الأمنة تعني احتمال انغلاق فقط.

Banker's Algorithm

Data structures

int n;                 // number of processes
int m;                  // number of resource types

int Available[m];       // Available[j] = k -> k free instances of resource j
int Max[n][m];          // Max[i][j] = k     -> Pi may request at most k of j
int Allocation[n][m];   // Allocation[i][j]  -> Pi currently holds k of j
int Need[n][m];         // Need[i][j] = Max[i][j] - Allocation[i][j]

Safety algorithm — isSafe()

bool isSafe() {
    int Work[m]    = copy(Available);
    bool Finish[n] = { false, false, ..., false };

    while (true) {
        int i = find index where Finish[i] == false AND Need[i] <= Work;
        if (no such i found)
            break;

        Work += Allocation[i];
        Finish[i] = true;
    }

    return all(Finish);   // true only if every process finished
}

Resource-request algorithm — requestResources(i, Request)

int requestResources(int i, int Request[m]) {
    if (Request > Need[i])
        return ERROR;          // process asked for more than its declared max

    if (Request > Available)
        return MUST_WAIT;      // not enough resources right now

    // pretend to allocate
    Available     -= Request;
    Allocation[i] += Request;
    Need[i]       -= Request;

    if (isSafe())
        return GRANTED;        // keep the new state

    // undo: state would be unsafe
    Available     += Request;
    Allocation[i] -= Request;
    Need[i]       += Request;
    return MUST_WAIT;
}

Deadlock Detection (single instance per resource)

Build a wait-for graph (processes only); Pi→Pj means Pi waits for Pj. Periodically search for a cycle — found cycle = deadlock. Cost: O(n²).
يُبنى رسم "الانتظار من أجل" (عمليات فقط)؛ Pi→Pj تعني انتظار Pi لـPj. يُفحص دوريًا بحثًا عن دورة؛ وجودها يعني انغلاقًا. التكلفة O(n²).

Chapter 9 — Main Memory (Part 1 + Part 2)

Address Binding

Compile time (fast, fixed location). Load time (relocatable code, bound at load). Execution time (process can move while running; needs base/limit registers; slow but flexible). Logical address (CPU-generated) vs Physical address (seen by memory unit) — same for compile/load binding, differ for execution-time binding.
زمن الترجمة (موقع ثابت، سريع)، زمن التحميل (كود قابل لإعادة التوضع)، زمن التنفيذ (يحتاج سجلات قاعدة وحد، بطيء لكن مرن). العنوان المنطقي يختلف عن الفعلي فقط في حالة الربط وقت التنفيذ.

MMU & Dynamic Loading/Linking

MMU: maps logical→physical at run time by adding a Relocation/Base register value to every address. Dynamic loading: a routine loads only when called. Dynamic linking: shared libraries linked at run time (Windows .DLL, Linux .so, macOS .dylib).
وحدة إدارة الذاكرة MMU تُترجم العنوان المنطقي للفعلي بإضافة قيمة سجل القاعدة. التحميل الديناميكي: لا يُحمَّل روتين حتى استدعائه. الربط الديناميكي: ربط مكتبات مشتركة وقت التشغيل.

Swapping

Swap time = (process size / transfer rate) × 2

Worked Example — Swap Time SOLVED

EN

Swap time = (100 / 50) x 2 = 2 x 2 = 4 sec = 4000 ms

حجم العملية 100 ميغابايت، سرعة القرص 50 ميغابايت/ثانية. زمن التبديل = (100÷50)×2 = 4 ثوانٍ. الضرب في 2 لأن العملية تُسحب وتُعاد مرتين.

Fragmentation & Hole-Fitting

External fragmentation — enough free memory total, but scattered (fixed by compaction or paging). Internal fragmentation — allocated block bigger than needed, waste inside the block itself. First-fit: first hole big enough. Best-fit: smallest hole big enough (least leftover). Worst-fit: largest hole (most leftover).
التجزّؤ الخارجي: مساحة كافية إجماليًا لكنها مبعثرة (يُحل بالدمج أو الترقيم). التجزّؤ الداخلي: كتلة أكبر من الحاجة فيضيع جزء داخلها. الملاءمة الأولى: أول فجوة كافية. الملاءمة الأفضل: أصغر فجوة كافية. الملاءمة الأسوأ: أكبر فجوة.

Paging — Visual Diagram

Paging removes external fragmentation by letting a process's logical memory be non-contiguous. Physical memory splits into fixed-size frames; logical memory splits into same-size pages. A page table maps page number → frame number.
Physical address = (Frame number × Frame size) + Offset
الترقيم يزيل التجزّؤ الخارجي بجعل الذاكرة المنطقية للعملية غير متجاورة. تُقسَّم الذاكرة الفعلية لإطارات ثابتة الحجم، والمنطقية لصفحات بنفس الحجم. جدول الصفحات يربط رقم الصفحة برقم الإطار. العنوان الفعلي = (رقم الإطار × حجم الإطار) + الإزاحة.

Memory Protection & Hardware Registers

Memory Stall: CPU access to main memory takes multiple cycles, causing the processor to stall. Fixed by hardware-managed cache.
Hardware Protection: Every logical address generated in user mode is checked by hardware: Base <= Address < Base + Limit. Any violation causes a TRAP to the OS.
توقف المعالج (Memory Stall): الوصول للذاكرة الرئيسية يستغرق دورات متعددة مما يجبر المعالج على التوقف؛ ويتم حلها عبر الذاكرة المخبئية (Cache).
حماية الهاردوير: يتم فحص كل عنوان منطقي عبر الهاردوير للتأكد من وقوعه في النطاق القانوني للعملية: Base <= العنوان < Base + Limit، وأي تجاوز يولد Trap للنظام لمنع تعديل بيانات النواة.

Page Table Implementation & TLB

Each process has its own page table; PTBR points to it (updated every context switch). Without help, every access = 2 memory accesses. The TLB caches page→frame mappings for near-instant lookup.
كل عملية لها جدول صفحات خاص؛ PTBR يُشير له ويتغير عند كل تبديل سياق. بدون تحسين، كل وصول يحتاج وصولين للذاكرة. الـTLB يخزّن مؤقتًا روابط صفحة→إطار لاستدعاء فوري تقريبًا.

Worked Example — Effective Access Time (EAT) SOLVED

EAT = (Hit ratio × hit time) + (Miss ratio × miss time)
Memory access = 10 ns. Hit = 10 ns. Miss = 20 ns. 
Hit ratio 80%: EAT = 0.80x10 + 0.20x20 = 12 ns   (20% slowdown)
Hit ratio 99%: EAT = 0.99x10 + 0.01x20 = 10.1 ns (only 1% slowdown)
AR
زمن الوصول للذاكرة 10 نانو. مع نسبة نجاح 80%: الزمن الفعلي = 12 نانو (تباطؤ 20%). مع نسبة نجاح 99%: الزمن الفعلي = 10.1 نانو (تباطؤ 1% فقط).

Valid/Invalid Bit, Hierarchical Paging, Shared Pages, Segmentation

Valid/invalid bit marks if a page belongs to the process's address space (access to "invalid" → trap). Hierarchical (two-level) paging pages the page table itself (address splits into p1, p2, offset) for very large address spaces. Shared pages: one physical frame referenced by multiple page tables (e.g. shared read-only code). Segmentation: like paging but based on the user's logical view (main program, functions, stack = variable-size segments); a segment table maps segment number → base + length.
بت الصلاحية يحدد إن كانت الصفحة جزءًا حقيقيًا من مساحة عنونة العملية (الوصول لصفحة غير صالحة يُسبب Trap). الترقيم الهرمي يُرقّم جدول الصفحات نفسه للمساحات الكبيرة جدًا. الصفحات المشتركة: إطار فعلي واحد تُشير له عدة جداول صفحات (كود مشترك للقراءة فقط). التجزيء: شبيه بالترقيم لكنه يعتمد على رؤية المستخدم المنطقية (برنامج رئيسي، دوال، مكدس = مقاطع متغيرة الحجم)؛ جدول المقاطع يربط رقم المقطع بعنوان القاعدة وطوله.
